# Project Governance

This document outlines the governance model for the **Wiki** repository.

## 🕊️ Overview
This project is a personal knowledge management system (PKM) maintained by **Suryakanta Mangaraj**. While it is primarily a personal project, it is open to community contributions and aims to be a high-quality public resource.

## 👑 Leadership
**Project Lead (Maintainer)**: [Suryakanta Mangaraj](https://github.com/suryakantamangaraj)

The Project Lead has the final authority on:
- Technical architecture and platform choice (Docusaurus).
- Content structure and domain categorization.
- Acceptance of contributions and Pull Requests.
- Enforcement of the Code of Conduct.

## 🤝 Decision Making
Decisions are generally made by the Project Lead. However, community input is highly valued and encouraged through:
- **GitHub Issues**: For discussing new features, content gaps, or bugs.
- **Pull Request Reviews**: For collaborative refinement of submitted content.

### Consensus Model
For significant changes (e.g., refactoring the entire sidebar or adding a major new domain), the Project Lead will open an issue for community feedback before proceeding. The goal is to reach a "lazy consensus" where silence implies consent after a reasonable time (typically 7 days).

## 🎖️ Contributor Roles
- **Contributors**: Anyone who submits an issue or a Pull Request.
- **Reviewers**: Trusted contributors who help review incoming PRs (appointed by the Project Lead).
- **Maintainers**: Individuals with write access to the repository (currently only the Project Lead).

## 📈 Evolution of Governance
As the repository grows and the community expands, this governance model may be updated to include more formal roles and voting processes.
